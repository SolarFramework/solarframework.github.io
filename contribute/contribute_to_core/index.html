SolAR Core presentation

The SolAR core handles

  • the dynamic load of different components embedded in containers downloaded from different distant servers.

  • the connection between some of these components, their configuration

  • the feature of saving the pose estimator scheme (the required components, their configurations as well as their connectivities)

  • to build a dynamic library that could be directly used by third applications.

  • multi-threading management (?) A VALIDER

  • local and distant distributed architecture management

Moreover, the *SolAR core* has to manage the compatibility between components licenses and supported  plateforms by informing the user of conflicting components due to incompatible license or plateform //availability.

Core SolAR is the main part of SolAR, where components can be connected to realize a pose estimation.

The whole solution for pose estimation, connecting different vision SolAR components is called a "pose estimation scheme" or a "pose estimation peipeline". Pose estimation scheme is currently done with the SLAM technique.

SLAM schema, since the Parallel Tracking And Mapping implementation, takes advantage of multi-threading to be able to execute the camera pose estimation as well as mapping and loop closure detection in real-time on low ressources terminals.
In this way, camera pose can be estimated at frame rate while mapping and loop closure detection that are processor-consuming can be executed in parallel when needed according to pre-defined criteria.

Moreover, augmented reality applications generally require very low latency tracking (even more when executed on see-through displays) requiring local computation, but mapping or loop closure detection can be remotely executed to reduce local terminal load.

SolAR Core repository structure

|- config |- doc |- interfaces |- libs |- models |- scripts |- src |- resources |- samples |- tests |- tools

config/

*application’s configuration (can be embedded in the application through a bundle or setup, or deployed in the OS' specific configuration folder - for instance /etc on unixes)

doc/

  • documentation

interfaces/

  • library public API (for C/C++ packages). Can also contain client/server remoting interface description files (Corba’s IDL, COM/DCOM, dbus' xml, C header file for in house stub/skeleton code generators for pcz etc…)

libs/

  • project directory entry point for all the submodules library the main project depends on (NOTE : submodules can also be included at other levels of the project tree, depending on their purpose)

model/

conception models (UML, Merise …)

scripts/

  • scripts (database creation, packages installation, deployment …) not used during project build

src/

main product/executable source code

resources/

main product/executable resources (deployed in /usr/share in linux, or bundled in the application)

samples/

  • usage samples

tests/

main product/executable’s tests

tools/

main product/executable’s tools used for project build(code generators …)

libs/libname/doc

  • libname’s documentation

libs/libname/interfaces/

libname’s public interface

libs/libname/src/

  • libname’s source code

libs/libname/samples/

  • libname’s use samples

libs/libname/tests/

  • libname’s tests
